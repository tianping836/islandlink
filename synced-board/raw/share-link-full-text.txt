屿连 IslandLink — 设计系统 v2

基于「连接」的品牌理念重建：海豚双游，人案相连。专业而温暖，精准而流动。

一、设计哲学

三条原则贯穿整个设计系统：

连接可见。 每到一个页面，用户应该一眼看到当前实体（人/案）的全部关联。关联不是藏在菜单里的功能，而是页面结构的一级公民。海豚头尾相接的意象落实为具体设计语言：每个实体的关联区域用环绕式布局，视觉上形成"相连"的闭环。

专业但不冷。 法律工具天然需要权威感和信任感，但"连接"的核心是人——法官、当事人、同行律师。色彩体系在传统法律蓝的基础上注入暖调，让每个角色类型有体温，让案件列表有呼吸感。不是冷冰冰的数据库，而是一个律师愿意每天打开的工作伴侣。

精准流动。 交互的节奏像海豚游泳——平滑、连续、不打断。导航转场用共享元素过渡，搜索补全用渐入而非弹跳，列表加载用淡入序列。每一个像素的运动都服务于"连接"这个核心隐喻：从一个实体到另一个实体的跳转，是游泳，不是跳跃。

二、色彩体系

2.1 品牌主色

以海豚双游为灵感，主色从深海蓝渐变到暖珊瑚，覆盖专业感和人情味两极。

色板 色值 用途 Ocean Deep（主色） #0D2137 导航栏、主要按钮、强调文字

iOS App 技术栈选型

决策日期：2026-06-16 状态：已确定

决策

SwiftUI 纯原生。不做跨平台，不用 Flutter / React Native。

约束条件

目标平台：仅 iOS，无 Android 需求

App 类型：工具 / 效率类

开发背景：非技术背景，依赖 AI 辅助

为什么是 SwiftUI

不需要跨平台，就没有理由承担跨平台的代价。 Flutter 和 React Native 的价值是"写一次双端运行"。只做 iOS 时，它们带来的性能折扣、原生能力受限、调试复杂度、苹果新特性滞后全是净损失。

SwiftUI 是所有 LLM 生成 Swift 代码的最佳目标。 学术评测和开发者实战一致确认：SwiftUI（声明式）的 AI 生成质量远高于 UIKit（命令式）。声明式框架与 AI 的"描述-生成"模式天然吻合——你描述界面长什么样，它直接渲染。UIKit 需要手动管理生命周期、约束和代理，AI 生成的代码编译通过率低，往往需要反复修 3-5 轮。

飞书多维表格 → 岛连 IslandLink：契合功能与借鉴方案

概述

飞书多维表格有 28 种字段类型、5 种视图类型、完整的关联引用系统。但岛连不是飞书——它是一个以人脉为中心的 Apple 原生应用，案件的定位是"极简记录"而非"全功能项目管理"。借鉴飞书不是为了复制它，而是从它的产品设计中提取那些符合岛连核心定位、能显著提升用户体验、实现成本可控的设计模式。

以下按契合度从高到低排列，每个功能点包含：与岛连的契合度评估、飞书的做法、岛连当前状态、借鉴方案、实现复杂度。

一、灵活字段的类型化（契合度：★★★★★）

飞书的做法

飞书的每个字段都有明确的类型编码——文本（1）、数字（2）、日期（4）、单选（7）、人员（11）、关联（18/21）等。用户创建字段时先选择类型，再填写值。类型决定了字段的输入方式（日期弹出 DatePicker）、显示样式（人员渲染为头像+姓名）、搜索行为（数字支持范围查询）和约束校验。

岛连当前状态

岛连已有灵活字段系统（FieldTemplate + CaseFieldValue），但所有字段值统一存为 String。这意味着"委托人"字段填的是纯文本"王强"，与系统中已有的 Person 记录没有任何关联。用户在案件页看到的是字符串，点不了也搜不到；在联系人页看不到自己作为委托人参与了哪些案件。

借鉴方案

给 FieldTemplate 增加 fieldType 枚举，区分四种类型：

飞书多维表格字段系统研究 — 对岛连 IslandLink 的启示

底层架构：一张编码表管 28 种类型

飞书多维表格的字段系统，底层是一套精心设计的类型编码 + UI 表现双层架构。它的 API 暴露了 23 种底层数据类型编码（从 type=1 的 Text 到 type=3001 的 Button），而通过 uitype 参数进一步分化出 28 种用户可见的字段类型——比如文本、条码、Email 三者底层共享编码 1（Text），数字、进度、货币、评分共享编码 2（Number）。这种"底层归一、表现分化"的设计让存储层保持结构简洁，而用户界面层可以灵活扩展出多种交互形态。

这套设计的哲学本质是：每个字段类型对应一个真实的业务语义，而非一个数据类型。 进度条不是"一个 0-100 的数字"，而是"一个需要视觉化展示的进度概念"；货币字段不是"一个带两位小数的数字"，而是"一笔需要显示币种符号的金额"。这给岛连的启示是——自定义字段系统不应该只提供"文本"和"日期"两个基础选择，而应该思考律师实际工作中的业务语义：委托人是一个"关联到人脉库"的字段，案号是一个"不可重复的唯一标识"，诉讼阶段是一个"可流转的状态"。

关联系统：飞书最值得借鉴的功能

飞书的双向关联（DuplexLink，type=21）是整个字段体系中最具差异化的设计。它不同于 Notion 简化的 Relation 二元模型，也不同于 Airtable 的 Linked Record + Lookup + Rollup + Count 四件套——飞书提供了单向关联（SingleLink，type=18）和双向关联两种模式，配合查找引用（Lookup，type=19）和公式（Formula，type=20），构成了一套完整的跨表数据引用链路。

这正是岛连天生的优势场景。岛连的核心是"人脉 ↔ 案件"的多对多关系——一个案件关联法官、当事人、对方律师等多个联系人；每个联系人也参与多个案件。目前岛连通过 CasePerson 枢纽表实现了这种关联，但在视图层面是单向的：你在案件页能看到关联的联系人，但在联系人页只能看到"参与 X 个案件"的计数。飞书的查找引用字段解决的就是这个问题：它允许你在关联表 A 的视图里，直接拉取关联表 B 的字段值并展示。

飞书 vs Airtable vs Notion：产品哲学的差异

三个产品在字段系统上的根本分歧反映了它们的产品定位。Airtable 是原生的关系型数据库，字段类型最严谨（30+ 种），支持 Rollup 聚合公式如 AVERAGE(values)，适合复杂数据建模。Notion 是"文档里的数据库"，数据库只是页面的属性集合，Relation + Rollup 的二元模型够用但简单，上限是 500 个属性。飞书走中间路线——关系能力接近 Airtable（双向关联是独有亮点），但又深度嵌入飞书生态（人员字段对接通讯录返回 id/name/email/enname 结构化对象、群组字段支持拉群、按钮字段可触发自动化），本质是一个低代码业务系统搭建平台。
